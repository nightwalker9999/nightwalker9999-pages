debugging-scenarios

Today I wanted to test myself.
Can I actually debug real problems without being told what to do?

Three scenarios. Hints needed. Evidence followed.
SCENARIO 01 — SERVICE DOWN

Cron was stopped. No hint on what was wrong — just "it's broken, fix it."

systemctl status cron → inactive (dead) → systemctl start cron

New PID. Back up. On-call engineers do this at 2am.

SCENARIO 02 — DISK AT 95%

500MB appeared from nowhere. Drilled from root down until I found it.

du -h -d 1 / → /tmp = 501M → du -h -d 2 /tmp/* → bigfile

One rm later. Disk back to 39%.

SCENARIO 03 — CONTAINER NOT RESPONDING

curl localhost:8080 failed. Nothing obvious. Started investigating.

ss -tulpn → nothing on 8080

Tried starting the container. Got: exec: --name: not found

Read the error. Flags were after the image name — Docker passed them to nginx instead. Fixed the order. Nginx responded.

The pattern across all three was the same:

→ Start at the service, not the network
→ Read the output, don't guess
→ Logs tell you everything if you know where to look

I needed hints to start. But once I had direction — I could follow the evidence myself.

That's what DevOps interviews test. Not whether you know the answer. Whether you know where to look.
